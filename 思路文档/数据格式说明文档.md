# 数据格式说明文档

## 1. 文档目标

本文档用于定义 EDA 原理图生成任务的数据集格式标准，作为后续程序化生成数据、质量校验、模型训练和结果导出的统一依据。

本任务当前采用的数据形式为：

- 输入：自然语言提示词 `prompt`
- 输出：结构化字典 `output`

本任务的核心不是让模型生成真实工程级原理图，而是让模型先学会：

- 从自然语言中识别电路功能
- 识别应包含的核心节点
- 生成合理的连接关系
- 生成规则化、统一化的元件坐标

因此，数据集设计必须满足以下原则：

- 强约束
- 易校验
- 可扩增
- 可转换
- 可程序化生成

## 2. 任务定义

### 2.1 输入

输入为一段自然语言提示词，用于描述一个基础功能电路的目标、所含元件和输出要求。

输入示例：

```text
设计一个完成 RC 低通滤波功能的 EDA 原理图，包含输入 VIN、电阻 R1、输出 OUT、电容 C1 和接地 GND。请给出所有节点的坐标以及节点之间的连接关系。
```

### 2.2 输出

输出为一个结构化字典，而不是直接输出图像，也不是直接输出自由文本说明。

标准输出应包含：

- `function`：电路功能名
- `components`：节点字典，记录每个节点的类型和坐标
- `edges`：连接字典，记录节点之间的连接关系
- 可选 `layout_type`：布局模板类型

输出示例：

```json
{
  "function": "RC低通滤波",
  "layout_type": "horizontal_main_vertical_branch",
  "components": {
    "VIN": {
      "type": "input",
      "x": 10,
      "y": 20
    },
    "R1": {
      "type": "resistor",
      "x": 40,
      "y": 20
    },
    "OUT": {
      "type": "output",
      "x": 70,
      "y": 20
    },
    "C1": {
      "type": "capacitor",
      "x": 70,
      "y": 50
    },
    "GND": {
      "type": "ground",
      "x": 70,
      "y": 80
    }
  },
  "edges": {
    "edge1": ["VIN", "R1"],
    "edge2": ["R1", "OUT"],
    "edge3": ["OUT", "C1"],
    "edge4": ["C1", "GND"]
  }
}
```

## 3. 为什么选择这种格式

相比直接使用如下格式：

```json
{
  "edge1": {"VIN": [10, 20], "R1": [40, 20]},
  "edge2": {"R1": [40, 20], "OUT": [70, 20]}
}
```

本规范更推荐“组件表 + 边表”结构，因为：

- 同一节点坐标只存一次，不会在不同边里重复出现
- 不容易出现同一个节点坐标不一致的问题
- 更适合程序检查、去重和纠错
- 更容易转换成图结构、网表结构或其他训练格式
- 更方便后续导出成你需要的 `edge1 -> 坐标对` 形式

因此，推荐做法是：

- 主数据集使用标准结构化格式
- 导出阶段再转换为展示型 edge 字典格式

## 4. 顶层字段定义

每条样本建议采用如下顶层结构：

```json
{
  "id": "sample_000001",
  "prompt": "...",
  "output": {
    "function": "...",
    "layout_type": "...",
    "components": {},
    "edges": {}
  }
}
```

字段说明如下。

### 4.1 `id`

样本唯一编号。

要求：

- 全局唯一
- 稳定可追踪
- 便于程序重建

推荐格式：

- `sample_000001`
- `rc_lowpass_0007`
- `led_indicator_0032`

### 4.2 `prompt`

自然语言提示词。

要求：

- 语义清晰
- 功能明确
- 元件完整
- 输出要求明确
- 用词尽量稳定

### 4.3 `output.function`

电路功能名。

要求：

- 使用标准功能名称
- 不混入多重复杂功能
- 不使用模糊表述

推荐值示例：

- `LED指示`
- `RC低通滤波`
- `RC高通滤波`
- `电阻分压`
- `电源去耦`
- `LC滤波`
- `上拉按键输入`
- `下拉按键输入`
- `二极管防反接`
- `简单整流滤波`

### 4.4 `output.layout_type`

布局模板类型，用于描述坐标生成规则。

推荐值示例：

- `chain_horizontal`
- `vertical_chain`
- `horizontal_main_vertical_branch`
- `split_branch`
- `power_load_ground_branch`

### 4.5 `output.components`

节点字典。键为节点名，值为节点详细信息。

推荐格式：

```json
{
  "R1": {
    "type": "resistor",
    "x": 40,
    "y": 20
  }
}
```

### 4.6 `output.edges`

连接字典。键为边名，值为长度为 2 的节点列表。

推荐格式：

```json
{
  "edge1": ["VIN", "R1"],
  "edge2": ["R1", "OUT"]
}
```

要求：

- `edge1` 到 `edgeN` 连续编号
- 每条边只连接两个节点
- 节点必须存在于 `components`
- 不允许引用未定义节点

## 5. 节点设计原则

### 5.1 节点是什么

在本数据集任务中，节点不是严格意义上的 EDA 软件内部 `net node`，而是任务级抽象后的“功能节点/元件节点/特殊电气节点”。

当前阶段，节点主要包括三大类：

- 元件节点
- 电源与地节点
- 输入输出与功能性节点

### 5.2 为什么不直接使用 net 节点

当前任务优先目标是让模型学会“功能需求到基础连接关系”的映射，而不是完全拟合真实 EDA 内部连接数据库。

因此当前阶段不单独建模：

- `N001`
- `Net_A`
- `Node_3`

原因如下：

- 会增加格式复杂度
- 会稀释模型对功能拓扑的学习
- 对当前小规模功能电路任务收益不大

后续若需要升级到更严格的网表表达，可以再扩展为：

- 元件节点
- 引脚节点
- 网络节点

## 6. 节点类型总表

以下是建议优先支持的节点类型集合。当前阶段不追求无限扩展，而是追求有限、稳定、可控。

### 6.1 电源与参考节点

| 节点类型 | 标准 type | 推荐命名 |
|---|---|---|
| 电源输入 | `power` | `VCC`, `VDD`, `VIN_PWR` |
| 地 | `ground` | `GND` |
| 输入电压节点 | `input` | `VIN` |
| 输出电压节点 | `output` | `OUT`, `VOUT` |
| 中间节点 | `junction` | `MID`, `NODE1` |

说明：

- `VCC` 和 `GND` 是最优先保留的电气参考节点
- `VIN` 更适合表示信号输入或电路输入端
- `OUT` 更适合表示输出端
- `MID` 用于分压或中间抽头

### 6.2 被动元件节点

| 节点类型 | 标准 type | 推荐命名 |
|---|---|---|
| 电阻 | `resistor` | `R1`, `R2` |
| 电容 | `capacitor` | `C1`, `C2` |
| 电感 | `inductor` | `L1`, `L2` |
| 负载电阻 | `load` | `LOAD`, `RL` |

说明：

- 若只是普通负载，可统一用 `LOAD`
- 若想体现是阻性负载，也可用 `RL`

### 6.3 半导体元件节点

| 节点类型 | 标准 type | 推荐命名 |
|---|---|---|
| 普通二极管 | `diode` | `D1` |
| 发光二极管 | `led` | `LED1` |
| 三极管 | `bjt` | `Q1` |
| MOS 管 | `mosfet` | `M1`, `Q1` |

说明：

- 当前阶段建议优先使用 `diode` 和 `led`
- `bjt`、`mosfet` 可在后续阶段加入

### 6.4 机电与控制节点

| 节点类型 | 标准 type | 推荐命名 |
|---|---|---|
| 开关 | `switch` | `SW1` |
| 按键 | `button` | `KEY1`, `BTN1` |

### 6.5 特殊功能节点

| 节点类型 | 标准 type | 推荐命名 |
|---|---|---|
| 输入端 | `input` | `VIN`, `SIG_IN` |
| 输出端 | `output` | `OUT`, `SIG_OUT` |
| 测试点 | `test_point` | `TP1` |

## 7. 节点命名规范

节点名必须稳定、规范、可程序解析。

### 7.1 总体要求

- 名称唯一
- 使用 ASCII 命名
- 名称中不使用空格
- 避免中英文混写
- 同类节点使用统一前缀

### 7.2 推荐前缀规范

| 类别 | 前缀 | 例子 |
|---|---|---|
| 电阻 | `R` | `R1`, `R2` |
| 电容 | `C` | `C1`, `C2` |
| 电感 | `L` | `L1` |
| 二极管 | `D` | `D1` |
| LED | `LED` | `LED1` |
| 三极管 | `Q` | `Q1` |
| MOS 管 | `M` 或 `Q` | `M1` |
| 开关 | `SW` | `SW1` |
| 按键 | `KEY` 或 `BTN` | `KEY1` |
| 输入 | 固定名 | `VIN` |
| 输出 | 固定名 | `OUT`, `VOUT` |
| 电源 | 固定名 | `VCC`, `VDD` |
| 地 | 固定名 | `GND` |
| 中间节点 | 固定名 | `MID`, `NODE1` |
| 负载 | 固定名 | `LOAD` |

### 7.3 命名禁忌

不建议使用：

- `电阻1`
- `电容A`
- `那个输出`
- `节点甲`
- `r1`
- `gnd`

原因：

- 不利于程序统一处理
- 容易产生大小写不一致
- 不利于后续模板复用

## 8. 输出字典的节点字段规范

`components` 中每个节点推荐包含以下字段：

```json
{
  "R1": {
    "type": "resistor",
    "x": 40,
    "y": 20
  }
}
```

字段说明：

- `type`：节点类型
- `x`：横坐标
- `y`：纵坐标

后续若需要扩展，可增加：

- `role`
- `orientation`
- `value`
- `group`

例如：

```json
{
  "R1": {
    "type": "resistor",
    "role": "series_resistor",
    "x": 40,
    "y": 20
  }
}
```

但当前阶段建议先保持最小必要字段，避免数据复杂度过高。

## 9. 边设计规范

### 9.1 边的定义

边表示两个节点之间的直接连接关系。

例如：

```json
"edge1": ["VIN", "R1"]
```

表示 `VIN` 与 `R1` 相连。

### 9.2 边的规则

- 每条边连接两个已定义节点
- 不允许自环，如 `["R1", "R1"]`
- 默认不允许重复边
- 边编号必须连续
- 边的顺序应尽量符合主链优先原则

### 9.3 推荐编号顺序

边的排序尽量按照以下顺序：

- 主链优先
- 从输入到输出
- 从电源到负载
- 最后再列支路和接地分支

例如 RC 低通：

```json
{
  "edge1": ["VIN", "R1"],
  "edge2": ["R1", "OUT"],
  "edge3": ["OUT", "C1"],
  "edge4": ["C1", "GND"]
}
```

这种顺序比随机排序更利于模型学习。

## 10. 提示词设计规范

提示词是输入端的关键设计点。当前任务不适合完全自由文本，而适合“固定模板 + 受控变体”的方式。

### 10.1 提示词必须包含的信息

每条提示词建议明确包含：

- 电路功能
- 核心元件
- 输出要求

可选包含：

- 电路用途
- 附加约束
- 节点命名要求

### 10.2 推荐提示词模板

模板 A：

```text
设计一个完成{功能}功能的 EDA 原理图，包含{元件列表}。请给出所有节点的坐标以及节点之间的连接关系。
```

模板 B：

```text
请设计一个用于{用途}的基础 EDA 原理图，电路中包含{元件列表}。请输出该电路的结构化结果，包括节点信息和连接关系。
```

模板 C：

```text
给定一个需要实现{功能}的简单电路，请使用{元件列表}构建 EDA 原理图，并返回标准字典格式的节点与边信息。
```

模板 D：

```text
生成一个实现{功能}的原理图样本，要求包含{元件列表}，并输出各节点坐标和 edge 连接关系。
```

### 10.3 提示词中“元件列表”的写法

建议明确列出节点名，而不是只说元件类别。

推荐：

```text
包含输入 VIN、电阻 R1、输出 OUT、电容 C1 和接地 GND
```

不推荐：

```text
包含一个电阻、一个电容和一个地
```

原因：

- 节点名明确，输出更稳定
- 更适合训练模型学习标准命名
- 更方便程序比对

### 10.4 提示词多样化策略

提示词不能完全一样，否则模型会机械记忆；但也不能过于自由，否则语义漂移。

建议变化范围包括：

- 句式变化
- 动词变化
- 功能描述变化
- 输出要求表述变化

建议不变化的部分：

- 节点名
- 功能核心含义
- 元件集合
- 输出目标

### 10.5 推荐的提示词变体来源

同一模板可以生成若干种提示词变体，例如 RC 低通：

变体 1：

```text
设计一个完成 RC 低通滤波功能的 EDA 原理图，包含输入 VIN、电阻 R1、输出 OUT、电容 C1 和接地 GND。请给出所有节点的坐标以及连接关系。
```

变体 2：

```text
请生成一个 RC 低通滤波电路的原理图样本，电路包含 VIN、R1、OUT、C1 和 GND。请输出标准字典格式的节点和边信息。
```

变体 3：

```text
构建一个简单的 RC 低通滤波 EDA 原理图，要求使用 VIN、R1、OUT、C1、GND 这些节点，并返回结构化的坐标与连接数据。
```

变体 4：

```text
给出一个实现 RC 低通滤波功能的基础原理图，节点包括 VIN、R1、OUT、C1 和 GND。请输出节点坐标和 edge 连接关系。
```

## 11. 建议优先支持的功能类别

第一阶段建议只覆盖高频、基础、标准的功能模板。

### 11.1 推荐第一阶段功能

- `LED指示`
- `RC低通滤波`
- `RC高通滤波`
- `LC滤波`
- `电源去耦`
- `电阻分压`
- `上拉按键输入`
- `下拉按键输入`
- `二极管防反接`
- `简单整流滤波`

### 11.2 为什么先做这些

因为这些模板具有共同特点：

- 功能明确
- 连接简单
- 节点少
- 画法稳定
- 易于程序布局
- 易于人工审核

## 12. 样本设计建议

### 12.1 每条样本的理想属性

每条样本应满足：

- 功能单一
- 结构完整
- 命名标准
- 坐标规则
- 边顺序稳定
- prompt 清晰

### 12.2 每类功能的样本数量建议

如果当前目标是先生成约 400 条高质量数据，建议：

- 10 个基础功能类别
- 每类 20 个拓扑或提示词变体
- 每类再做 2 种左右布局微扰版本

这样可得到：

- 10 x 20 x 2 = 400 条

### 12.3 先生成模板，再生成数据

建议流程如下：

1. 先定义金模板
2. 再为模板生成提示词变体
3. 再根据布局规则生成坐标
4. 再导出边字典
5. 最后做自动校验

## 13. 坐标设计原则

### 13.1 坐标不是随机噪声

坐标必须体现原理图布局规则，而不是纯随机。

错误做法：

- 每个节点完全随机坐标
- 同类模板布局风格完全不同
- GND 在顶部，VCC 在底部
- 节点重叠

### 13.2 推荐布局原则

- 主链尽量水平排列
- 支路尽量垂直引出
- GND 优先放在下方
- VCC/VIN 优先放在左侧或上方
- 输出优先放在右侧
- 坐标落在统一网格上

### 13.3 推荐坐标参数

建议：

- 起点坐标取 10 的整数倍
- 水平步长 `dx` 取 20、30、40
- 垂直步长 `dy` 取 20、30、40
- 所有坐标均为整数

## 14. 数据存储格式建议

推荐最终存储为 JSONL，每行一条样本：

```json
{"id":"sample_000001","prompt":"...","output":{"function":"...","layout_type":"...","components":{},"edges":{}}}
{"id":"sample_000002","prompt":"...","output":{"function":"...","layout_type":"...","components":{},"edges":{}}}
```

原因：

- 适合程序流式生成
- 适合训练集读取
- 适合后续过滤和追加

## 15. 导出格式建议

虽然主存格式推荐使用“组件表 + 边表”，但可以导出为展示型 edge 字典格式。

导出示例：

```json
{
  "edge1": {
    "VIN": [10, 20],
    "R1": [40, 20]
  },
  "edge2": {
    "R1": [40, 20],
    "OUT": [70, 20]
  }
}
```

建议：

- 主数据保留标准结构
- 训练前如确有需要再转换
- 不建议把导出格式作为唯一数据源

## 16. 质量校验规则

数据集生成后必须经过自动校验。

### 16.1 结构校验

- `prompt` 不为空
- `function` 不为空
- `components` 非空
- `edges` 非空
- 每条边引用的节点必须已定义

### 16.2 命名校验

- 节点名唯一
- 符合命名规范
- 大小写统一

### 16.3 坐标校验

- 坐标必须为整数
- 不允许节点重叠
- 同一节点在整条样本中坐标唯一

### 16.4 功能校验

- 必备节点存在
- 连接关系符合模板
- 不出现无关节点

## 17. 当前阶段不建议纳入的数据内容

为了保证第一版数据质量，当前阶段不建议加入：

- 复杂多级放大器
- 运放反馈网络
- MCU 外围系统
- 总线通信原理图
- 多电源域系统
- 引脚级细粒度网表
- 大量真实工程器件参数

原因是这些会显著提高生成复杂度和校验成本，不适合第一版数据集。

## 18. 推荐的生成工作流

建议采用以下流程。

1. 定义基础功能类别
2. 为每类功能编写金模板
3. 为每个模板定义节点清单、连接清单和布局类型
4. 使用提示词模板程序化生成 `prompt`
5. 使用布局规则程序化生成坐标
6. 自动生成 `components` 与 `edges`
7. 导出 JSONL
8. 运行自动质检
9. 筛除异常样本

## 19. 一句话结论

本项目当前最合适的数据集设计不是“自由 prompt 对自由 edge 字典”，而是：

- 受控提示词
- 标准化节点命名
- 标准化组件表
- 标准化边表
- 规则化坐标
- 可导出的 edge 表示

只有这样，数据才能真正支撑后续模型稳定学习，而不是让模型去学习大量格式噪声和表达漂移。
